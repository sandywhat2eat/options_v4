# Real-Time Options Monitoring System Guide

## Overview

This guide explains the new real-time monitoring system that uses Dhan WebSocket market feeds and Supabase realtime subscriptions to provide sub-second trade monitoring and exit execution.

## System Architecture

### Core Components

1. **WebSocket Manager** (`core/websocket_manager.py`)
   - Manages Dhan live market feed connections
   - Handles price caching with Redis + in-memory fallback
   - Auto-reconnection and error handling
   - Event-driven price notifications

2. **Supabase Realtime** (`core/supabase_realtime.py`)
   - Subscribes to trades table changes
   - Instant notifications for new positions
   - Database change streaming
   - Connection management with heartbeat

3. **Position Cache Manager** (`core/position_cache_manager.py`)
   - In-memory position tracking
   - Real-time P&L calculations
   - Fast position lookups
   - Database synchronization

4. **Realtime Market Fetcher** (`data_scripts/realtime_market_fetcher.py`)
   - Unified price interface
   - WebSocket first, REST fallback
   - Batch and individual quote support
   - Automatic instrument discovery

5. **Realtime Automated Monitor** (`realtime_automated_monitor.py`)
   - Main monitoring system
   - Event-driven exit evaluation
   - Immediate order execution
   - Comprehensive logging

## How It Works

### 1. System Initialization

```
[Startup] → WebSocket Manager → Supabase Realtime → Position Cache → Monitor Loop
```

1. **WebSocket Connection**: Connects to Dhan live market feed
2. **Realtime Subscription**: Subscribes to trades table changes
3. **Position Loading**: Loads all open positions into cache
4. **Price Subscription**: Subscribes to price feeds for all instruments
5. **Monitor Start**: Begins event-driven monitoring

### 2. Price Update Flow

```
Dhan WebSocket → Price Cache (Redis/Memory) → Position Update → Exit Evaluation → Order Execution
```

**Event Sequence:**
1. Price update received via WebSocket
2. Cache updated (Redis + in-memory)
3. Positions recalculated with new prices
4. Exit conditions evaluated
5. Orders placed if triggers met
6. Database updated with results

### 3. Exit Trigger Evaluation

The system evaluates multiple exit conditions:

#### Profit Targets
- **Primary Target**: e.g., 50% profit
- **Scaling Targets**: 25%, 50%, 75% partial exits
- **Trailing Stops**: Dynamic profit protection

#### Stop Losses
- **Percentage-based**: e.g., -50% of maximum loss
- **Absolute Value**: e.g., ₹5,000 loss limit
- **Time-based**: Maximum holding period

#### Time Exits
- **DTE Thresholds**: e.g., close at 7 days to expiry
- **Theta Decay**: Accelerated time decay triggers
- **Expiration Management**: Pre-expiry closure

### 4. Order Execution Process

```
Exit Signal → Order Validation → Dhan API → Order Confirmation → Database Update
```

1. **Signal Generation**: Exit condition met
2. **Order Validation**: Check position size, market hours
3. **API Call**: Place market order via Dhan
4. **Confirmation**: Wait for order fill confirmation
5. **Database Update**: Record exit details and P&L

```python
# In realtime_automated_monitor.py
WEBSOCKET_CONFIG = {
    'auto_reconnect': True,
    'max_reconnect_attempts': 5,
    'reconnect_delay': 5,  # seconds
    'heartbeat_interval': 30,
    'price_cache_ttl': 300  # 5 minutes
}
```

### Monitoring Intervals

```python
# Event-driven (instant) + periodic checks
MONITORING_CONFIG = {
    'price_update_immediate': True,  # WebSocket events
    'position_sync_interval': 30,    # Database sync
    'health_check_interval': 60,     # System health
    'cleanup_interval': 300          # Cache cleanup
}
```

### Exit Thresholds

```python
# Configurable per strategy
EXIT_CONFIG = {
    'profit_target': 0.5,      # 50% profit
    'stop_loss': -0.5,         # 50% of max loss
    'time_exit_dte': 7,        # Close at 7 DTE
    'urgency_thresholds': {
        'HIGH': 0.8,           # 80% of stop loss
        'MEDIUM': 0.6,         # 60% of stop loss
        'NORMAL': 0.4          # 40% of stop loss
    }
}
```
